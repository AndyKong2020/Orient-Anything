# Orient Anything NPU 部署报告

生成日期：2026-06-16

## 任务概述

Orient Anything V1 是一个面向单张图像的物体朝向估计项目，用于解决“给定一张包含单个主要物体的图片，判断观察者相对于物体的三维朝向”这一问题。项目输入为 RGB 图像，输出为四类结果：azimuth 表示水平面观察方位，polar 表示观察高度角，rotation 表示视角旋转角，confidence 表示图像中的物体是否具有明确方向性。交互式页面还会把预测出的三维坐标轴叠加到原图上，方便直观看到模型判断的物体朝向。

项目方法由三部分组成。第一步是图像预处理，可选择是否先做背景移除，以减少背景干扰。第二步是用 DINOv2-Large 视觉 backbone 提取图像特征，再通过一个 MLP 预测头输出离散角度分布和方向置信度。第三步是把模型输出解码为角度数值，并用 CPU 侧图像渲染逻辑生成坐标轴叠加图。项目还提供测试时增强策略，通过多次随机裁剪投票提升鲁棒性，但默认交互式推理路径不启用该策略。

本次部署目标是在 Ascend NPU 环境中复现项目原有功能，而不是重新设计应用或替换 WebUI。部署重点是让 DINOv2-Large 与 MLP 预测头在 NPU 上完成模型前向，同时保留原始单图推理、背景移除、坐标轴叠加和 Gradio 交互能力。由于项目没有自带数据集级评测脚本，本次验证以官方 V1 在线 Demo 行为对齐、单图功能 smoke、NPU tensor 运行和 WebUI/API 可用性为主。

## 部署改造

部署使用独立 Python 环境承载项目依赖，并复用 Ascend PyTorch 运行栈。关键组件如下：

| 组件 | 版本 |
| --- | --- |
| Python | 3.11.6 |
| torch | 2.10.0+cpu |
| torch-npu | 2.10.0 |
| transformers | 4.38.0 |
| huggingface_hub | 0.26.5 |
| gradio | 5.9.0 |
| rembg | 2.0.69 |
| onnxruntime | 1.26.0 |
| numpy | 1.26.4 |
| pillow | 10.2.0 |
| matplotlib | 3.11.0 |
| scikit-image | 0.26.0 |

NPU 改造主要集中在设备选择、张量流转、模型路径配置和服务启动参数四个方面。

| 改造方向 | 改造内容 | NPU 目的 |
| --- | --- | --- |
| 设备选择 | 推理启动时优先检测 Ascend NPU，NPU 可用时将模型移动到逻辑 `npu:0`；否则保留 CUDA/CPU 回退逻辑。 | 保持原项目跨设备可运行，同时在 Ascend 环境中默认使用 NPU 模型前向。 |
| 可见卡隔离 | 通过 Ascend 运行时可见卡配置只暴露一张后四卡中的可用卡，进程内映射为 `npu:0`。 | 满足只使用后四张卡的约束，避免误占前四张卡或其他任务设备。 |
| 输入张量迁移 | 图像预处理仍在 CPU 侧完成，生成的 `pixel_values` 明确迁移到 NPU。 | 让 DINOv2-Large backbone 和 MLP 预测头在 NPU 上执行。 |
| 输出解码 | 模型输出先从 NPU 显式转回 CPU，再执行切片、`argmax`、`softmax` 和标量转换。 | 避免 NPU tensor 与 CPU/NumPy/PIL 后处理混用导致设备不一致，同时保持解码结果稳定。 |
| 输出头兼容 | 解码逻辑同时支持当前官方 V1 Demo 使用的 902 维预测头，以及旧示例中可能出现的 722 维预测头。 | 降低更换 checkpoint 或沿用旧示例时的适配风险。 |
| 模型缓存配置 | 模型和处理器缓存目录可通过环境变量配置到数据盘。 | 避免大模型缓存落到系统盘，同时让容器内服务可重复启动。 |
| WebUI 服务启动 | 原始 Gradio 服务保留，监听地址和端口改为可配置；本地可达性检查绕过代理。 | 保留项目原有用户体验，并支持通过端口转发在本机浏览器访问。 |

需要特别说明的是，本次没有把背景移除、图像渲染、PIL/NumPy 图像叠加或 Gradio 服务改造成 NPU 算子路径。这些部分是项目交互体验的一部分，但不是模型主干计算。当前 NPU 部署的核心边界是：模型前向在 NPU，输入预处理和结果可视化仍在 CPU。

## 功能验证结果

本次验证围绕“项目原有功能是否在 NPU 部署后可用”和“核心数值输出是否与官方 V1 Demo 对齐”展开。验证方式分为三类：直接数值断言、接口返回结果检查、官方 Demo 对比。产物包括模型输出角度、置信度、WebUI 返回图像和测试日志。

| 验证功能 | 验证用例 | 验证方式 | 验证产物 | 结果 |
| --- | --- | --- | --- | --- |
| NPU 基础运行 | 在隔离后的单张 NPU 上创建 tensor 并求和。 | 直接运行 NPU tensor smoke，检查 `torch_npu` 可导入、NPU 可用、设备数量和计算结果。 | `torch_npu 2.10.0`，NPU 可用，设备数为 1，tensor 求和结果为 `2.0`。 | 通过。 |
| 输出头解码兼容性 | 构造 902 维和 722 维预测张量，并在 NPU 上执行解码。 | 标准库 unittest，对 azimuth、polar、rotation、confidence 和未知维度报错做断言。 | 2 个测试用例通过，用时 2.390s。 | 通过。 |
| 直接模型推理，无背景移除 | 使用项目内置示例图，关闭背景移除，直接调用模型推理。 | 加载 DINOv2-Large 与 Orient Anything 预测头，在 NPU 上前向，CPU 解码输出。 | azimuth `3.0`，polar `0.0`，rotation `1.0`，confidence `0.9965`，耗时 0.077s。 | 通过。 |
| 直接模型推理，开启背景移除 | 使用同一示例图，开启背景移除后再推理。 | 背景移除在 CPU/ONNXRuntime 完成，模型前向在 NPU 完成，CPU 解码输出。 | azimuth `355.0`，polar `0.0`，rotation `2.0`，confidence `0.2945`，耗时 0.553s。 | 通过。 |
| 原始 WebUI/API，无背景移除 | 通过 Gradio 接口上传同一示例图，关闭背景移除和测试时增强。 | 调用原始 WebUI 暴露的推理 API，检查接口返回的图像产物和文本结果。 | 返回坐标轴叠加图；文本输出为 azimuth `3.0`，polar `0.0`，rotation `1.0`，confidence `1.0`。 | 通过。 |
| 原始 WebUI/API，开启背景移除 | 通过 Gradio 接口上传同一示例图，开启背景移除，关闭测试时增强。 | 调用原始 WebUI 推理 API，检查背景移除后结果图和文本输出。 | 返回背景处理后的坐标轴叠加图；文本输出为 azimuth `355.0`，polar `0.0`，rotation `2.0`，confidence `0.29`。 | 通过。 |
| 官方 V1 Demo 对齐 | 使用同一示例图分别调用官方 V1 Demo 和本地 NPU WebUI。 | 对比两边返回的角度和置信度文本，不做主观目视判断作为唯一依据。 | 关闭背景移除时官方与本地均为 `3.0, 0.0, 1.0, 1.0`；开启背景移除时官方为 `355.0, 0.0, 2.0, 0.34`，本地为 `355.0, 0.0, 2.0, 0.29`。 | 核心角度对齐；背景移除路径 confidence 有小幅差异。 |

验证结论如下：

1. NPU 运行时、模型加载、模型前向和输出解码均已跑通。
2. 关闭背景移除时，本地 NPU 结果与官方 V1 Demo 完全一致。
3. 开启背景移除时，三项角度输出与官方 V1 Demo 一致，confidence 存在小幅差异。该差异来自背景移除和图像预处理链路的后端差异，不影响模型主体功能复现。
4. WebUI 产物包含返回图像和角度文本。图像产物用于确认交互链路完整，最终正确性判断以接口返回的数值结果和官方 Demo 对比为准。

## Benchmark 与指标说明

当前项目没有随仓提供独立 benchmark、批量 eval 脚本或带标注的公开评测集入口。因此，本次部署不能给出数据集级 MAE、accuracy、角度误差分布或论文指标复现结果。项目原始使用方式主要面向单图交互推理和 Python 调用，而不是标准化批量评测。

本次可量化的复现指标包括：

| 指标类型 | 本次结果 | 说明 |
| --- | --- | --- |
| NPU 基础可用性 | NPU tensor smoke 通过 | 验证 Ascend PyTorch 运行栈可用。 |
| 输出头解码回归 | 2 个 unittest 通过 | 覆盖当前和旧式预测头形状。 |
| 单图直接推理 | 背景移除开/关均通过 | 验证模型前向与数值解码。 |
| WebUI/API 功能 | 背景移除开/关均通过 | 验证原始交互入口、返回图像和文本输出。 |
| 官方 V1 Demo parity | 角度结果对齐 | 验证部署后行为与官方在线 Demo 一致。 |

如果后续需要做论文级或生产级 benchmark，需要补充以下内容：固定评测集、标注格式、角度误差定义、方向置信度阈值策略、背景移除开关策略、是否启用测试时增强、批量推理脚本和结果汇总方式。没有这些输入时，不应把单图 smoke 扩展解释为完整指标复现。

## 部署结论

Orient Anything V1 已在 Ascend NPU 环境完成原始功能复现。部署后的核心链路是：CPU 侧完成图像预处理，NPU 侧完成 DINOv2-Large 与 MLP 预测头前向，CPU 侧完成输出解码、背景移除后处理和坐标轴可视化。这个边界符合项目原有交互式推理形态，也避免把非模型部分误写成 NPU 算子能力。

从项目功能看，单图朝向估计、可选背景移除、角度文本输出、置信度输出和坐标轴叠加图均可用。从 NPU 适配看，模型主干和预测头已能在 NPU 上稳定完成前向，输出结果与官方 V1 Demo 在核心角度上对齐。当前部署适合作为单卡交互式演示、NPU 亲和性分析和后续批量评测改造的基础。

当前不建议宣称该项目已经实现“全链路 NPU 原生化”。背景移除、图像渲染、WebUI 服务和部分图像处理仍是 CPU/ONNXRuntime/Python 路径。若要进一步提升 NPU 利用率，应优先增加批量推理入口，并把性能统计限定在模型前向部分，再单独统计预处理和后处理耗时。
